# Linux GPU Start Guide

This guide is for moving Steer-SAO onto a Linux box with NVIDIA RTX 6000 cards and
starting real SA3 MuseControlLite-style adapter experiments.

## 1. What We Have Already

- SA3-native adapter code for `stabilityai/stable-audio-3-small-music`.
- CLI commands for fetching HF audio-text datasets, preparing control features, training adapters, and generation.
- A sample config at `configs/train_all_controls.yaml`.
- CPU-safe tests that validate the local scaffolding without downloading gated model weights.

Nothing has been trained yet. The repo currently contains code only, not adapter weights.

## 2. Machine Setup

Use Linux with recent NVIDIA drivers, CUDA-compatible PyTorch, Python 3.10+, and enough local disk.

```bash
git clone git@github.com:manoskary/Steer-SAO.git
cd Steer-SAO

python3.10 -m pip install uv
uv sync --extra dev --extra data --extra train
```

Authenticate with Hugging Face after accepting the gated SA3 model terms:

```bash
uv run huggingface-cli login
```

Keep `HF_TOKEN` in your shell or HF cache only. Do not commit tokens.

Quick checks:

```bash
uv run python -c "import torch; print(torch.cuda.is_available(), torch.cuda.get_device_name(0))"
uv run python -m unittest discover -s tests -v
```

## 3. Data: What We Need

We need audio-caption pairs. The controls themselves are generated by this repo from the audio:

- melody: chroma-like time-varying features
- rhythm: onset/energy curves
- dynamics: loudness curve
- audio: SAME latent reference audio

You do not manually annotate these controls. You only need local audio files and text captions.

For a first smoke test, use 20-50 clips and intentionally overfit. For useful adapters, expect hundreds
to thousands of clips at minimum. For all controls together, more data is better.

## 4. Recommended Dataset Path

Best quick-start HF option:

- [`mrfakename/cc0-music-captioned`](https://huggingface.co/datasets/mrfakename/cc0-music-captioned)
  - HF shows Audio + Text columns, parquet format, about 8.68k estimated rows, and about 294 GB total.
  - The dataset card says it was collected from CC0 instrumental music sites and all songs were human-created.
  - This is the cleanest bootstrap choice because the audio is already present on HF and the captions are in a simple `text` column.

Fetch a small subset first:

```bash
uv run steer-sao hf-manifest \
  --dataset mrfakename/cc0-music-captioned \
  --split train \
  --out data/cc0_music/train_50.jsonl \
  --audio-dir data/cc0_music/audio \
  --limit 50
```

Then prepare control features:

```bash
uv run steer-sao prepare-data \
  --manifest data/cc0_music/train_50.jsonl \
  --out cache/controls/cc0_music_50 \
  --controls melody_stereo,rhythm,dynamics
```

The command writes `cache/controls/cc0_music_50/manifest.controls.jsonl`.
Point `configs/train_all_controls.yaml` at that prepared manifest before training.

## 5. Other Dataset Options

- [`google/MusicCaps`](https://huggingface.co/datasets/google/MusicCaps): excellent captions, but the HF dataset is a CSV with YouTube IDs and timestamps rather than bundled audio. Use mostly for evaluation or research prototyping after checking audio rights and availability.
- [`M2UGen/MUCaps`](https://huggingface.co/datasets/M2UGen/MUCaps): larger music-caption set with audio archives and generated captions. Useful after validating license and quality for your use case.
- MTG-Jamendo / Song Describer: closest to the original MuseControlLite lineage. The MTG-Jamendo page links Song Describer captions for 706 permissively licensed recordings. Good for evaluation and small experiments; setup is less one-command than the CC0 HF dataset.
- Your own licensed dataset: best long-term path. Use stems, production music, Jamendo/FMA/FreePD-style sources, or internal music where you can legally train.

Avoid using unclear scraped music datasets for publishable or commercial work.

## 6. Training

Edit `configs/train_all_controls.yaml`:

```yaml
manifest: cache/controls/cc0_music_50/manifest.controls.jsonl
output_dir: checkpoints
controls:
  - melody_stereo
  - rhythm
  - dynamics
  - audio
training:
  max_steps: 1000
  batch_size: 1
  learning_rate: 0.0001
```

Start with one GPU and a tiny overfit run:

```bash
CUDA_VISIBLE_DEVICES=0 uv run steer-sao train --config configs/train_all_controls.yaml
```

Expected first goal: loss decreases on a tiny dataset and a checkpoint appears under `checkpoints/`.

## 7. Generate With A Trained Adapter

```bash
uv run steer-sao generate \
  --prompt "warm instrumental synthpop, 120 BPM, bright drums and nostalgic chords" \
  --duration 30 \
  --adapter checkpoints/mcl_sa3_all_controls_step_001000.safetensors \
  --control-audio data/cc0_music/audio/00000000.wav \
  --controls melody_stereo,rhythm,dynamics \
  --out generated_audio/smoke.wav
```

For audio-reference conditioning too:

```bash
uv run steer-sao generate \
  --prompt "ambient electronic instrumental with spacious pads" \
  --duration 30 \
  --adapter checkpoints/mcl_sa3_all_controls_step_001000.safetensors \
  --control-audio data/cc0_music/audio/00000000.wav \
  --controls melody_stereo,rhythm,dynamics,audio \
  --out generated_audio/smoke_audio_control.wav
```

## 8. Practical RTX 6000 Notes

- Start with `batch_size: 1` and `duration <= 30` for smoke tests.
- Increase duration and batch size only after the whole pipeline works.
- Keep generated WAVs, data, caches, and checkpoints out of git; `.gitignore` already covers them.
- Use separate runs for attribute-only and audio-control adapters if joint training sounds unstable.
- The first serious milestone is not FAD or benchmark quality; it is obvious overfit control adherence on a tiny set.

## 9. Current Gaps

- We have the preprocessing/preparing pipeline, but only basic feature extraction is implemented.
- We have training code, but it has not yet been exercised on a real CUDA SA3 environment.
- We do not yet have curated train/validation manifests or trained adapter checkpoints.
- After the first GPU smoke run, inspect generated audio and decide whether to improve feature extractors, add validation metrics, or split control-specific adapter training.
